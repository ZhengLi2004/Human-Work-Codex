# Skill: Work — Research Report to TODOs

## 目标

把人类提供的完备研究报告忠实映射为当前项目中的一个或多个可执行 TODO。Work 负责宏观思路与实现之间的抽象衔接，不重新设计人类研究。

## 输入

- 人类完整研究报告及明确修订；
- 当前项目代码、数据说明、已有结果和相邻任务；
- TODO 与工作流模板；
- `AGENTS.md` 和适用 Skills。

## 权威边界

人类报告中的研究问题、研究假设、初步研究方案、研究对象和证据方向是上位输入。Work 可以：

- 把抽象对象映射到项目对象；
- 暴露项目不兼容点；
- 将方案拆成大小适中的证据闭环；
- 形式化变量、控制、指标、重复和完成证据；
- 建议阶段路线。

Work 不得静默替换研究问题、方法语义或科学目标。

## 工作步骤

### 1. 保持来源可追溯

每个 TODO 引用人类报告路径、版本和对应章节。任务目标保持原意，并明确它只承接报告中的哪一部分。

### 2. 映射项目事实

写明：

- 抽象对象对应的项目数据、模块和行为；
- 方法组成对应的既有能力和待实现能力；
- 指标如何从项目输出获得；
- 哪些前提已确认，哪些交给 Exploration；
- 哪些项目差异可能改变方案含义，需要人类决定。

### 3. 拆分 TODO

按“一个主要研究问题 + 一组可独立审查的实现与证据”拆分，而不是按文件、函数或阶段拆分。避免一个 TODO 同时承载多个需要独立科学决策的方向。

### 4. 形式化实验设计

提取或明确：

- 任务变量和全部计划取值；
- 控制变量及不一致处理；
- 完整组合集合；
- 指标定义；
- 独立重复单位和每组合重复数；
- 指标在单次重复内计算、再跨重复汇总的聚合层级；
- 配对、阻断或分层关系；
- 不完整与不可比条件。

未规定且实验含随机性时，默认每组合至少 3 次；若资源压力显著，标为待人类批准，不自行降低。

### 5. 写技术手段，不写执行手册

TODO 只描述项目映射、技术手段、变量、指标、任务边界和完成证据。阶段流程、测试方法、checkpoint、日志和资源规则由 AGENTS、Skills 与工作流承担。

### 6. 初始化 Codex 调用

- 初始化 workflow 和建议路线；
- 状态设为 `awaiting_human_todo_approval`；
- 写出 Codex 首个阶段的精确范围；
- 向人类提交 TODO 供批准；
- 人类批准后分配整个 TODO 给 Codex；

## 输出

- 一个或多个 `TODO.md`；
- 每个 TODO 对应的 `workflow.yaml`；
- 简短任务拆分说明；
- 待人类批准项。

不得编写实现代码，不得虚构项目事实。
